Executive Summary

AkibaLens는 아키하바라, 돈키호테, 중고 피규어샵에서 박스 없이 진열된 피규어를 발견했을 때 정확한 캐릭터명, 작품명, 상품명을 몰라 검색과 가격 비교가 어려운 문제를 해결하기 위해 만든 AI 쇼핑 보조 MVP이다.

사용자는 피규어 사진을 업로드하고, 시스템은 OpenAI Vision으로 캐릭터와 작품을 추정한 뒤 OpenAI Web Search를 통해 상품 후보를 검색한다. 이후 후보를 다시 점수화하고, 일본 쇼핑몰 검색 링크와 가격 확인 preview를 제공한다.

Problem Definition

User Pain

  • 피규어 이름을 모른다.
  • 캐릭터명과 작품명을 모른다.
  • 일본어 상품명을 검색하기 어렵다.
  • 제조사와 라인업을 구분하기 어렵다.
  • 현재 가격이 합리적인지 판단하기 어렵다.

MVP Goal

  • 사진 한 장으로 검색 시작점을 만든다.
  • 캐릭터/작품을 우선 식별한다.
  • 상품 후보와 근거를 함께 제공한다.
  • 일본 쇼핑몰 검색 링크를 제공한다.
  • 불확실한 정보는 단정하지 않는다.

System Architecture

Key Technical Decisions

Why API-first?

프로젝트의 목적은 커스텀 모델 성능 경쟁이 아니라, 실제 사용자가 원하는 워크플로우인지 빠르게 검증하는 것이었다. 따라서 CLIP fine-tuning이나 자체 CNN 학습은 MVP 범위에서 제외했다.

Why Search-based Retrieval?

Vision만으로 상품명과 라인업을 확정하기 어렵다는 문제가 있었다. 이를 보완하기 위해 Vision 결과를 검색어로 변환하고, Web Search로 상품 후보를 가져온 뒤 다시 reranking하는 구조를 선택했다.

Project Origin and Planning

이 프로젝트는 “AI 기술을 보여주기 위한 데모”가 아니라, 실제 쇼핑 경험에서 반복적으로 겪은 문제를 제품으로 풀어보려는 시도에서 시작했다.

일본 여행 중 아키하바라, 돈키호테, 중고 피규어샵을 방문하면 박스 없이 진열된 피규어를 자주 보게 된다. 마음에 드는 피규어를 발견해도 캐릭터명, 작품명, 제조사, 라인업, 일본어 상품명을 모르면 검색 자체가 막힌다. 특히 입문자나 일본어 검색에 익숙하지 않은 여행객에게는 가격 비교와 구매 판단이 더 어렵다.

Initial Problem Statement

  • 사진은 있는데 상품명을 모른다.
  • 일본어 검색어를 만들기 어렵다.
  • 동일 캐릭터의 유사 상품이 많아 가격 비교가 어렵다.
  • 중고샵 가격이 합리적인지 현장에서 판단하기 어렵다.

Initial Product Hypothesis

  • 사용자는 정확한 정답보다 검색 시작점이 먼저 필요하다.
  • 캐릭터/작품 식별만으로도 검색 난이도를 크게 낮출 수 있다.
  • 일본 쇼핑몰 링크가 함께 있으면 구매 판단이 빨라진다.
  • 불확실성을 명시하면 오답도 탐색 후보로 활용할 수 있다.
Planning Principle: 2-4주 내에 실제로 만질 수 있는 MVP를 만드는 것을 우선했다. 자체 CV 모델 학습, CLIP fine-tuning, 결제, 구매대행, 위시리스트, 가격 알림은 MVP 이후로 미뤘다.

MVP Scope Definition

MVP 범위는 “사진 업로드 → 피규어 식별 → 상품 후보 검색 → 쇼핑몰 링크 제공 → 가격 비교 preview”로 제한했다. 정확한 가격 자동 수집이나 자체 모델 학습은 제품 검증 이전 단계에서는 과하다고 판단했다.

Detailed MVP Pipeline

AkibaLens의 핵심 파이프라인은 이미지 자체를 웹에서 직접 검색하는 구조가 아니다. Vision 모델이 이미지를 텍스트 단서로 바꾸고, 그 텍스트 단서를 검색 및 후보 비교에 활용한다.

1. User Upload JPG/PNG/WEBP 이미지 검증, 미리보기, 8MB 제한, upload state 관리.
→
2. Vision Identification 캐릭터, 작품, 시각 특징, 불확실한 필드, 검색어를 구조화된 JSON으로 반환.
→
3. Retrieval Query 일본어/영어 marketplace-ready query 생성. 캐릭터/작품/포즈/색상 중심.
4. Web Search OpenAI Web Search로 Good Smile, AmiAmi, Mandarake, Suruga-ya 등 후보 검색.
→
5. Candidate Reranking 검색 후보 제목/스니펫과 Vision 단서를 비교해 score, evidence, concerns 생성.
→
6. User Decision UI 상품 후보, 쇼핑몰 링크, 가격 preview, 불확실성, 다음 확인 방법을 함께 표시.
Key Design: 이 파이프라인은 “AI가 정답을 맞혔다”가 아니라 “사용자가 검색과 비교를 시작할 수 있게 만들었다”에 초점을 둔다. 그래서 confidence, unknown fields, candidate concerns를 UI에 노출한다.

Engineering Incidents and Resolutions

개발 과정에서 단순 구현보다 중요한 부분은 막혔던 문제를 정의하고, 대안을 비교하고, 현재 MVP에 맞는 결정을 내리는 과정이었다.

Incident Impact Resolution Reasoning
OpenAI quota / billing issue Vision API 연결은 되었지만 실제 분석 요청이 실패했다. API credit을 충전하고 같은 endpoint를 재검증했다. 무료/결제 상태가 API 사용 가능성과 다르다는 점을 확인했다.
Vision output too conservative 캐릭터/작품 대신 시각 묘사만 반환되어 검색 결과가 무의미했다. 캐릭터/작품은 적극 추정하고, 제조사/라인업은 보수적으로 두도록 프롬프트를 수정했다. 사용자는 정답 확정보다 검색 시작점이 필요했기 때문이다.
Web ChatGPT vs API quality gap 웹 ChatGPT는 상품명을 맞추는데 API는 틀리는 현상이 있었다. Vision 단독 호출과 검색/후보 검증이 결합된 흐름의 차이로 분석했다. API 모델이 “쓰레기”라기보다 도구/검색/프롬프트/후보 검증 차이가 컸다.
Google Custom Search 403 검색 API 기반 RAG 구현이 막혔다. OpenAI Web Search provider를 기본으로 전환했다. 신규 고객 접근 제한 가능성이 있어 MVP 일정상 우회가 합리적이었다.
DuckDuckGo/Bing HTML search instability 무료 검색 실험은 challenge 또는 낮은 후보 품질 문제가 있었다. 실험 코드는 참고로 두고 안정 provider 중심으로 전환했다. MVP에서는 무료보다 재현성과 결과 품질이 중요했다.
Filename label leakage 샘플 파일명에 nendoroid/luminasta 같은 정답 힌트가 있어 평가가 과대평가됐다. 파일명을 sample_01.png 형태로 바꾸고 재평가했다. AI 평가에서 입력 leakage를 제거해야 신뢰 가능한 결과를 얻을 수 있다.
Production returned mock mode Render 배포 후 프론트는 동작했지만 실제 OpenAI가 아니라 mock 응답을 반환했다. 환경변수 이름을 OPENAI_API_KEY로 수정하고 manual deploy했다. 배포 환경에서 env var 이름 하나가 전체 AI 연결을 바꿀 수 있음을 확인했다.
Long inference time Vision + Web Search + reranking 조합으로 응답이 길어졌다. 식별 단계 web search는 기본 비활성화하고, 검색 결과 수를 3개로 제한했다. 정확도와 속도 사이에서 MVP는 “작동하는 후보 탐색”을 우선했다.

What the Experiment Proved

  • 캐릭터/작품 식별은 MVP 가치 검증에 충분히 유효하다.
  • 정확한 상품명/라인업 식별은 별도 문제이며 난이도가 훨씬 높다.
  • 파일명, 박스 텍스트, 가격표 등 작은 힌트가 모델 결과에 큰 영향을 줄 수 있다.
  • Vision이 틀리면 Web Search도 틀린 방향으로 진행된다.
  • 따라서 UI는 단일 정답보다 후보와 불확실성을 보여줘야 한다.

Product and UX Lessons

What Worked

  • 사진 업로드 후 캐릭터/작품을 추정하는 흐름은 직관적이었다.
  • 일본어 검색어를 자동 생성하는 기능은 사용자 문제와 잘 맞는다.
  • 쇼핑몰 링크가 있으면 사용자가 바로 검증할 수 있다.
  • candidate reranking은 검색 후보를 단순 나열보다 이해하기 쉽게 만든다.

What Needs Work

  • 응답 시간이 길어 사용자가 멈춘 것처럼 느낄 수 있다.
  • 결과 화면이 정보량이 많아 단계별 UX가 필요하다.
  • 상품 라인업 정확도는 아직 불안정하다.
  • 가격 비교가 preview라 실제 구매 판단에는 한계가 있다.

다음 UX 개선은 “식별 중 → 상품 후보 검색 중 → 가격 확인 링크 생성 중”처럼 분석 단계를 분리해 보여주는 staged UI가 적절하다. 이렇게 하면 느린 추론 시간을 사용자가 이해할 수 있고, 중간 결과도 더 빨리 전달할 수 있다.

Vision output was too conservative for loose figures Prompt changed to aggressively provide character/series hypotheses while keeping manufacturer/line uncertain.
Google Custom Search access failed for new project Switched MVP retrieval provider to OpenAI Web Search.
Filename hints distorted evaluation Renamed samples to neutral filenames and reran evaluation.
Production backend returned mock mode Fixed Render environment variable name to OPENAI_API_KEY and redeployed.

Current Limitations

No true reverse image search Exact product line unstable Price parsing not implemented Render cold start OpenAI API cost

AkibaLens는 현재 “정확한 상품 확정 서비스”가 아니라 “사진을 검색 가능한 상품 후보와 일본어 검색어로 바꿔주는 AI 검색 보조 MVP”로 포지셔닝하는 것이 가장 적절하다.

Limitation Why It Matters Possible Fix
추론 시간이 길다 사용자는 버튼을 누른 뒤 앱이 멈춘 것처럼 느낄 수 있다. staged response, streaming status, retrieval 비동기화, 검색 후보 수 조정
UI 정보량이 많다 초보 사용자는 candidate, evidence, unknown fields를 한 번에 이해하기 어렵다. 상단에는 가장 유력한 후보, 하단에는 상세 근거를 접을 수 있게 구성
정확도 편차 Reze 케이스처럼 비슷한 캐릭터로 오식별되면 검색도 함께 틀어진다. 추가 이미지 입력, 후보군 다중 유지, 상품 이미지 비교
상품 라인업 식별 Grandista, Luminasta, Ichiban Kuji처럼 세부 라인업은 이미지 단독으로 어렵다. 박스/받침대/로고 OCR, 상품 DB/RAG, marketplace image matching
가격 정보 현재 가격 preview는 실제 가격 파싱이 아니므로 구매 판단에는 제한이 있다. 사이트 정책 검토 후 가격 parser 또는 official API/affiliate feed 탐색
운영 비용 OpenAI Vision + Web Search + reranking은 요청당 비용과 latency가 있다. 캐싱, rate limit, lower-cost model routing, search result reuse

Conclusion

AkibaLens MVP는 배포된 환경에서 이미지 업로드, OpenAI Vision 분석, 검색 후보 생성, candidate reranking, 쇼핑몰 링크 제공까지 end-to-end로 동작한다.

포트폴리오 관점에서 이 프로젝트의 가치는 단순히 AI API를 호출했다는 점이 아니라, 실제 사용자 문제에서 시작해 MVP 범위를 정하고, 평가 과정에서 label leakage와 상품 식별의 난이도를 발견했으며, 이를 바탕으로 불확실성을 전제로 한 제품 UX를 설계했다는 점이다.
